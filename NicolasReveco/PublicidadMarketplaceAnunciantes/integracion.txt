=====================================================================
 BOUNDED CONTEXT 6: PublicidadMarketplaceAnunciantes  (puerto 5006)
 Reglas de integración (solo documentación; NO se implementan las llamadas HTTP)
=====================================================================

RESPONSABILIDAD
---------------
Gestiona el marketplace de publicidad: anunciantes, campañas, creativos, targeting,
inventario, entrega de anuncios, medición (impresiones/clics) y reporting.
Referencia contenido/canales por UUID solo para targeting; no almacena metadata del contenido.


1. DEPENDENCIAS SALIENTES (lo que este contexto necesita consumir de otros)
--------------------------------------------------------------------------
- Descubrimiento y Personalización (puerto 5003):
    GET http://localhost:5003/api/trending?page=1&pageSize=20
    GET http://localhost:5003/api/search?q=...
    -> Para enriquecer el contexto de targeting (tendencias e intereses) al decidir
       qué anuncio servir. La decisión local usa los campos Countries/Interests del
       Targeting; Descubrimiento aporta señales adicionales opcionales.

- Catálogo Editorial y Derechos (puerto 5002):
    GET http://localhost:5002/api/catalogitems/{catalogItemId}
    GET http://localhost:5002/api/channels/{channelId}
    -> Para validar que el contenido/canal donde se mostrará el anuncio existe y es
       monetizable (visibilidad pública, sin restricciones que impidan publicidad).

- Monetización del Ecosistema Creador (puerto 5005):
    POST http://localhost:5005/api/revenue
    Body ejemplo:
      {
        "channelId": "<uuid-del-canal>",
        "videoId": "<uuid-del-catalogItem>",
        "source": "Ads",
        "amount": 0.85
      }
    -> Tras medir eventos facturables (RF-F7), el gasto de la campaña sobre un video se
       traduce en ingreso para el creador. Publicidad NOTIFICA el ingreso a Monetización.


2. DEPENDENCIAS ENTRANTES (lo que este contexto expone para que otros consuman)
------------------------------------------------------------------------------
- GET http://localhost:5006/api/ads/serve?placement=PreRoll&country=CL&interest=gaming
    -> Consumido por el reproductor / Publicación y Distribución al iniciar una
       reproducción, para obtener la decisión de anuncio (campaignId + creativo).
       Respuesta 404 con code "NO_FILL" si no hay campaña elegible.

- GET http://localhost:5006/api/reporting/campaigns/{id}
    -> Consumido por dashboards de anunciantes (RF-F8).

- POST http://localhost:5006/api/measurement/events
    -> Consumido por el reproductor para reportar impresiones y clics (RF-F7).

Otros endpoints expuestos (gestión por el anunciante):
  /api/advertisers, /api/campaigns (+ activate/pause/complete),
  /api/campaigns/{id}/creatives, /api/campaigns/{id}/targeting, /api/inventory.


3. ORDEN DE LLAMADAS EN FLUJOS TÍPICOS
--------------------------------------
A) Alta y activación de campaña:
   1. POST /api/advertisers                      (RF-F1)
   2. POST /api/campaigns                         (RF-F2, Draft)
   3. POST /api/campaigns/{id}/creatives          (RF-F3)
   4. PUT  /api/campaigns/{id}/targeting          (RF-F4)
   5. POST /api/campaigns/{id}/activate           (RF-F2, -> Active)

B) Entrega + medición durante una reproducción:
   1. (Reproductor) GET /api/ads/serve            (RF-F6) -> { campaignId, creative }
   2. (Reproductor) POST /api/measurement/events  (RF-F7, Impression) -> Spend += costo
   3. (Reproductor) POST /api/measurement/events  (RF-F7, Click)
   4. (Publicidad) POST a Monetización /api/revenue (liquida ingreso al creador)
   5. (Anunciante) GET /api/reporting/campaigns/{id} (RF-F8)

C) Contexto cruzado para mostrar un video con anuncio:
   cliente -> Descubrimiento (feed/búsqueda)
           -> Catálogo (metadata + visibilidad)
           -> Publicación (URL de streaming)
           -> Publicidad (GET /api/ads/serve para el pre-roll)


4. MANEJO DE CONSISTENCIA EVENTUAL
----------------------------------
- Si Monetización (5005) NO responde al reportar un ingreso:
  el evento de medición YA quedó registrado localmente (fuente de verdad del gasto).
  La notificación de ingreso se encola para reintento idempotente; Monetización debe
  reconciliar por (campaignId, eventId) para evitar doble conteo. No se pierde dinero
  porque el gasto vive en Publicidad y el ingreso se deriva de él.

- Si Descubrimiento/Catálogo NO responden durante la decisión de anuncio:
  /api/ads/serve degrada con elegancia usando solo los campos locales de Targeting;
  si aun así no hay campaña elegible, responde 404 NO_FILL (sin anuncio), nunca un error 500.

- Presupuesto agotado durante la entrega:
  al registrar un evento cuyo costo lleva Spend >= Budget, la campaña pasa a Completed
  automáticamente y deja de ser elegible en /api/ads/serve. Puede existir una pequeña
  sobre-entrega (over-delivery) por eventos en vuelo; se acepta como inconsistencia
  temporal acotada.

- Contenido despublicado:
  si el video donde se sirvió el anuncio se despublica (Catálogo devuelve 404/Private),
  los eventos de medición ya contabilizados se conservan, pero no se sirven nuevos
  anuncios sobre ese contenido.
